# Skill: Fill Proposal Template

This skill is called when generating a Word document proposal from parsed RFP data.
It is NOT used for the text proposal (generate-proposal.md handles that).

## Purpose

Convert the structured `parsed` RFP object into a formatted Word (.docx) proposal letter
following the WATT Consulting Group proposal template structure.

## Template Structure

The proposal document is generated by `buildProposalDoc.js` and follows this order:

### 1. Recipient Block (variable)
- Company name (bold)
- Street address
- City, Province, Postal Code
- Attn: [Name], [Title]
- Blank line
- Re: [projectName] – RFP No. [rfpNumber]
- Date (current date, YYYY-MM-DD)
- Our File No: [YEAR]-XX   ← always leave as XX, never fill in

### 2. Opening Paragraph (mostly fixed)
- "Thank you for the opportunity to provide a proposal for the Civil Engineering services
  associated with the [projectName] project."
- "The following outlines our company profile, understanding of the project, our proposed
  team, and our work plan and fees."

### 3. WE ARE WATT (fixed boilerplate)
Company description. Do not modify.

### 4. PROJECT UNDERSTANDING (variable)
Use `parsed.summary` if available.
Otherwise write 2–3 sentences from `parsed.projectName`, `parsed.location`,
`parsed.buildingSize`, `parsed.siteAreaHectares`, and any other project details.
Keep it factual — no fluff.

### 5. SCOPE OF WORK (fully variable — comes from parsed.phases)

For each phase in `parsed.phases`:
- Use `phase.name` exactly as the subheading
- List deliverables as bullet points: `d.description`
- Skip deliverables flagged `not_applicable` or `recommend_remove`
- Do NOT include flag labels in the document — just the description text
- `coordinate_only` items are already rewritten as "Coordinate and contribute to..."

Phase order follows the RFP order exactly.

### 6. Assumptions (variable — comes from parsed.assumptions)
Standard opening sentence, then bullet list of `parsed.assumptions`.
If empty, fall back to the default assumption list in buildProposalDoc.js.

### 7. Our Team (fixed boilerplate)
Tibor Tuss, Zach Bridger, Jeremy Laverdiere bios. Do not modify.

### 8. FEES (variable — phase names from parsed.phases, amounts TBD)
Table with one row per active phase + Total row.
All amounts are "TBD" until the fee calculator (Excel template) is integrated.
Note: "All fees are exclusive of GST. Fees are valid through December 31, [year]."

### 9. Closing + Signature (fixed)
Jeremy Laverdiere signature block.

## Key Rules

- NEVER include flag labels (coordinate_only, not_applicable, recommend_remove) in output
- NEVER fill in the file number — always leave as YEAR-XX
- Date is always today's date in YYYY-MM-DD format
- Do not modify or summarize the team bios or company description sections
- Phases with ALL deliverables flagged not_applicable/recommend_remove are excluded entirely

details from the RFP: `parsed.clientName`, and ask the user for the contact name if unknown.
